1. Practice Safe Clicks: Enable MFA Everywhere

If you do only one thing from this list, make it this one. Multi-Factor Authentication (MFA) adds a crucial second layer of security. A password can be stolen, but a physical token or an app on your phone is much harder to compromise.

  • The Golden Rule: Your root account‚Äîthe email address you used to create your AWS account‚Äîis the superuser with the keys to the entire kingdom. It must have MFA enabled. No exceptions.

Default password settings are often too permissive. You can, and should, set a strong password policy for all your IAM users to prevent weak or easily guessable passwords.

In the IAM dashboard, you can configure a policy that requires:

  • Minimum Length: At least 12-14 characters.
  • Character Complexity: A mix of uppercase letters, lowercase letters, numbers, and symbols (e.g., !@#$%).
  • Password Expiration: Require users to change their passwords periodically (e.g., every 90 days).
  • Prevent Reuse: Stop users from recycling their last few passwords.

Your developers use access keys (an Access Key ID and a Secret Access Key) for programmatic access to AWS via the CLI, SDKs, and other tools. These long-lived credentials are powerful and, if leaked, pose a massive security risk.

  • Establish a Rotation Policy: Just like passwords, access keys should be rotated regularly. A 90-day rotation is a strong industry standard.

The root user has god-mode privileges. It can do anything, including deleting all your resources and closing the account. For this reason, you should never use it for your daily work.

  1. Enable MFA on the root user (see point #1).
  2. Create a separate IAM user for yourself and grant it administrative privileges by attaching the AWS-managed AdministratorAccess policy.
  3. Log out of the root user and log back in with your new admin IAM user.
  4. Lock away your root user credentials in a secure place.

Only use the root user for a very small, specific set of tasks that require it, like changing your AWS Support plan.

  • Attach policies to the Groups: Attach finely-tuned IAM policies to each group that grant only the necessary permissions for that role.
  • Add users to the Groups: When a new developer joins, simply add them to the Developers group.

6. Let Your Services Do the Talking: Use IAM Roles

Your applications running on EC2, Lambda, or ECS often need to access other AWS services (like S3 or DynamoDB). The worst way to grant this access is by hardcoding access keys into your code or configuration files.

The secure, modern approach is to use IAM Roles for Services:

  1. Create an IAM Role that has the specific permissions your application needs (e.g., ‚Äúallow read/write access to my-cool-bucket‚Äù).
  2. Attach that role to the AWS service where your code is running (e.g., launch your EC2 instance or configure your Lambda function with this role).

The service then automatically receives temporary, rotated credentials. There are no static keys to steal from your code or your servers. This is a non-negotiable best practice.

  • Your IAM users exist only in the main management account.
  • In your Production account, you create IAM Roles (e.g., Production-Admin-Role).
  • Your users in the main account use the Switch Role feature to temporarily gain access to the Production account.

This provides a powerful security boundary and a crystal-clear audit trail, all managed from a single location.
